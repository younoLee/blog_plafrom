# 블로그 만들기 #10 — 내 사이트 침투점검하고 구멍 막기

> 웹/클라우드 입문자가 읽어도 이해되게 — '보안 점검을 어떻게 하고, 찾은 걸 어떻게 실제로 막는가'의 전 과정.

## 0. 오늘 한 일 한눈에

라이브로 돌아가는 내 사이트를 보안 관점으로 통째로 뜯어봤다. 코드·인증·업로드·인프라·CI를 다 훑고(정적 감사), 사이트엔 해 없는 요청만 보내 설정을 확인하고(비침투 정찰), 쓰는 라이브러리를 알려진 취약점 목록과 대조했다(의존성 스캔). 그 결과 '진짜 위험한 것'을 찾아 코드·인프라로 막고, 라이브에 배포·검증까지 했다.

- 전체 감사: 인증·SSRF·XSS·권한·업로드·인프라 — 즉시 악용 가능한 치명 취약점은 없었음(그간 여러 번 감사한 덕)
- 진짜 구멍: 로그인 없이도 서버를 멈출 수 있는 DoS(라이브러리 취약점 2건) → 버전 올려 막고 배포
- 공개 저장소에 새어나간 AWS 배포 키 → 아예 없애고 'OIDC'라는 키 없는 인증으로 교체
- XSS 2차 방어선(CSP 헤더) 추가 — 안전하게 Report-Only로 검증 후 실제 적용
- 오리진(CDN→서버) 평문 구간 하나는 도메인이 없어 보류 — '리스크 수용'으로 문서화

> **🔎 비유** 도둑처럼 문·창문을 직접 흔들어본 게 아니라(그건 운영 사이트라 위험), 설계도와 자물쇠를 꼼꼼히 뜯어보고 약한 걸 찾아 바꿔 단 날이다. 그러다 새로 안 사실 하나 — 대문 열쇠 사본이 동네 게시판(공개 GitHub)에 붙어 있었다.

## 1. '터는 것'과 '점검'의 차이 — 뭘 어떻게 했나

"털어줘"는 원래 진짜 공격(침투 테스트)을 뜻한다. 그런데 이건 실제로 돌아가는 운영 사이트라, 함부로 공격을 날리면 진짜 인증 메일이 나가고, AI 호출 비용(실제 돈)이 들고, 최악엔 사이트가 멈춘다. 그래서 해 없는 방법으로 했다.

**① 정적 감사** — 코드를 통째로 읽어 약점을 찾는다(백엔드 2316줄 + 프론트 + terraform + CI). 실제로 공격하지 않고 '구조상 위험한 곳'을 짚는 방식.

**② 비침투 정찰** — 라이브에 해 없는 요청만 보내 응답 헤더·열린 포트·저장소 공개여부 등 '설정'을 확인. 예: 보안 헤더가 붙어 있는지, 서버 포트가 외부에 열렸는지.

**③ 의존성 스캔** — 내가 쓰는 라이브러리 버전을 '알려진 취약점 목록(CVE)'과 대조. 내 코드가 멀쩡해도 갖다 쓴 부품에 구멍이 있을 수 있다.

> **🔎 비유** 은행을 진짜 털어보는 대신(위험), 도면·자물쇠 모델·CCTV 배치를 검토하고 '이 자물쇠는 이런 결함이 있다'를 목록과 대조한 것.

> **🛠 전문가 노트** 진짜 능동 공격이 필요하면 '운영 사이트'가 아니라 '로컬에 똑같이 띄운 복제본'을 치는 게 정석이다 — 메일은 가짜 서버(Mailpit)가 받고, 비용도 0, 멈춰도 상관없다. 오늘은 거기까진 안 갔고, 위 3종으로 충분히 실제 구멍을 잡았다.

## 2. 제일 위험했던 것 — 로그인 없이 서버를 멈추는 DoS

내 코드보다 '갖다 쓴 부품'에 구멍이 있었다. 파일 업로드 같은 '폼 데이터'를 해석하는 라이브러리 두 개가 알려진 취약점 버전이었다.

**원인 ①** — 폼 파싱 부품(python-multipart)이, 악의적으로 만든 요청을 받으면 데이터를 1바이트씩 처리하며 로그를 남발해 서버의 처리 흐름을 통째로 멈춘다.

**원인 ②** — 또 다른 부품(starlette)은 '파일 이름이 없는 폼 칸'을 크기 제한 없이 메모리에 쌓는다. 큰 걸 몇 개만 동시에 보내도 메모리가 꽉 차 서버가 죽는다.

**왜 심각(무인증)** — 우리 서버는 요청의 '본문(폼)'을 먼저 해석한 다음에야 '로그인 검사'를 한다. 즉 로그인도 안 한 사람이 업로드 주소에 악성 요청만 던져도, 로그인 거부(401)가 뜨기 전에 이미 파서가 돌아 서버가 멈춘다. 게다가 서버가 가장 작은 사양(t2.micro)이라 넘기기도 쉽다.

**수정** — 세 라이브러리를 패치 버전으로 올렸다(python-multipart·starlette, 그리고 함정 때문에 프레임워크 FastAPI도). 올린 뒤 실제로 설치해 앱이 정상 로드되는지, 라이브 재배포 후 사이트가 멀쩡한지 눈으로 확인.

> **🔎 비유** 택배 상자를 '열어보기도 전에' 포장 테이프를 한 겹 한 겹 떼느라 직원이 다른 일을 못 하게 만드는 장난이다. 게다가 신분증 확인(로그인)이 그 뒤라서, 회원이 아니어도 아무나 걸 수 있었다.

> **🛠 전문가 노트** 코드만 읽었을 땐 '업로드는 승인된 사람만 되니 저위험'이라 넘길 뻔했다. 그런데 라이브러리 버전을 취약점 목록과 대조하고, 프레임워크가 '언제 로그인 검사를 하는지' 순서를 확인하니 '무인증'으로 재평가됐다. 가정을 실제로 확인하는 게 감사의 핵심이다. 참고로 라이브러리를 올릴 때 프레임워크가 '옛 버전만 허용'하도록 묶여 있어 충돌났고, 프레임워크도 호환 버전으로 같이 올려 맞물렸다.

## 3. 동네 게시판에 붙은 열쇠 사본 — 공개 저장소의 AWS 키

감사 중 저장소의 기록 파일에서 'AWS 배포 키 ID'가 커밋된 걸 발견했다. 그리고 이 GitHub 저장소가 '공개(public)'였다.

**다행** — 열쇠에서 진짜 위험한 절반(비밀 키)은 새지 않았다 — 그건 GitHub 금고(Secrets)에만 뒀더라. 그래서 이 ID 만으론 문을 못 연다 = 직접 악용은 불가. 하지만 공개된 열쇠 사본은 없애는 게 정석.

**근본 해결(로테이션 대신 OIDC)** — 새 키를 또 만들어 두는 대신 '키 없는 인증(OIDC)'으로 바꿨다. 배포 로봇(GitHub Actions)이 올 때마다 AWS에 '나 이 저장소 맞다'를 증명하고, '몇 분 뒤 자동 소멸하는 일회용 출입증'을 받아 쓰게 했다. 그런 다음 노출됐던 장기 키를 아예 삭제.

> **🔎 비유** 영구 비밀번호가 적힌 열쇠를 복사해 여기저기 두는 대신, 올 때마다 '5분 뒤 스스로 사라지는 일회용 출입증'을 발급받게 바꾼 것. 잃어버릴 영구 열쇠가 없으니 유출될 것도 없다.

> **🛠 전문가 노트** 노출이 '키 ID만'이라 급하진 않았다. 그런데 어차피 OIDC로 가면 장기 키가 필요 없어지니 '로테이션(새 키 발급)'은 버리는 일이 된다 → 곧장 OIDC로 가고 옛 키는 삭제가 가장 깔끔했다. 배포가 새 방식으로 성공(초록)한 걸 확인한 뒤에 키를 지웠다(fallback 확보).

## 4. 설령 뚫려도 — XSS 2차 방어선(CSP)

지금 글·댓글 화면은 위험한 스크립트를 이미 막고 있다(마크다운 라이브러리 기본 방어). 그래도 '혹시 나중에 구멍이 나도' 막는 2차선으로 CSP(콘텐츠 보안 정책) 헤더를 추가했다. '이 사이트는 우리 것 외의 스크립트는 실행 금지' 같은 규칙을 브라우저에 알려주는 것.

**방식** — 우리 CDN(CloudFront)이 무료 요금제라 정식 방법이 막혀 있어, 작은 함수를 붙여 헤더를 주입했다. 처음엔 'Report-Only'(사이트는 안 깨지고 위반만 기록) 모드로 배포해 실제로 문제없는지 확인한 뒤, 진짜 차단 모드로 전환.

> **🔎 비유** 낯선 사람이 가게에 들어오더라도 계산대·금고엔 손 못 대게 미리 규칙을 붙여둔 것. 그리고 규칙을 '진짜 시행' 전에 '경고만' 모드로 돌려보고, 정상 손님이 안 걸리는지 확인하고 켠 셈이다.

> **🛠 전문가 노트** 보안 헤더는 잘못 걸면 멀쩡한 사이트가 하얗게 죽는다. 그래서 Report-Only로 먼저 관측 → 확인 → enforce가 프로덕션 정석이다. 빌드 결과물에 인라인 스크립트가 없음을 미리 확인해 스크립트를 '우리 것만'으로 엄격히 잠갔다.

## 5. 오늘 못 막은 것 하나 — 그리고 그게 왜 괜찮은가

CDN(CloudFront)에서 우리 서버(EC2)로 가는 구간이 암호화가 안 돼 있다(평문). 이건 오늘 못 막고 '리스크 수용'으로 남겼다.

**왜 보류** — 이 구간을 암호화하려면 '신뢰된 인증서'가 필요하고, 그건 '내 도메인'이 있어야 발급된다. 지금 도메인이 없다(기본 CloudFront 주소만 씀). 도메인은 유료고, 대안(로드밸런서)은 월 비용 함정이라 프로젝트 규칙상 제외.

**왜 그나마 괜찮나** — 그 구간은 이미 방화벽(보안그룹)으로 'CloudFront만 접속 가능'하게 잠겨 외부인은 못 들어온다. 평문이 흐르는 건 AWS 내부망뿐이라 실질 위험이 낮다. 나중에 커스텀 도메인을 붙일 때 HTTPS까지 함께.

> **🔎 비유** 가게에서 창고로 가는 '사내 전용 복도'가 암호화 안 된 상태다. 근데 그 복도는 외부인 출입금지라 우리 직원만 지나간다 — 완벽하진 않아도 급하진 않다.

## 6. 오늘의 삽질 & 배운 것 (진짜 배운 건 여기)

**SSH가 안 들어가짐** — 재배포하려는데 서버 접속이 안 됐다. 원인 둘 — ① 서버 OS(Amazon Linux)의 접속 계정이 'ubuntu'가 아니라 'ec2-user'였고, ② 키 파일이 명령어가 찾던 폴더가 아니라 다른 곳(~/.ssh/)에 있었다. 헛다리도 짚었다(내 점검 도구가 특정 포트를 막아 '방화벽 문제'로 오인 → AWS에 직접 물어보고 정정).

**git이 없다?** — 서버에 'git' 명령이 없었다 = 애초에 코드를 git으로 받은 게 아니라 '파일 복사(scp)'로 올린 것. 그래서 바뀐 파일만 scp로 올려 재빌드했다.

**버전 핀 함정** — DoS 고치려 라이브러리를 올렸더니 '프레임워크가 옛 버전만 허용'이라 충돌. 프레임워크도 호환 버전으로 같이 올려 해결.

> **🛠 전문가 노트** '됐겠지'로 안 넘기고 매번 실제로 확인한 게 오늘의 핵심이다. 설치가 되는지, 앱이 뜨는지, 라이브가 멀쩡한지, 배포가 초록인지, 키가 정말 삭제됐는지 — 전부 눈으로 확인하고 넘어갔다.

## 7. 오늘 익힌 핵심 개념

- 감사 ≠ 침투: 운영 사이트엔 해 없는 방법(정적 감사·비침투 정찰·의존성 스캔)으로, 진짜 공격은 로컬 복제본에
- 취약점은 '내 코드'만이 아니라 '내가 쓰는 라이브러리'에도 산다 → 버전 대조(의존성 스캔)는 필수 습관
- 가정을 확인하라: '승인된 사람만'이라 믿었는데 실제 실행 순서를 보니 무인증이었다
- 유출 대응의 근본은 '없애기': 장기 키를 새로 발급(로테이션)하기보다 OIDC로 키 자체를 제거
- 방어는 여러 겹: 이미 XSS를 막고 있어도 CSP로 한 겹 더(설령 뚫려도)
- 공개 저장소의 대가: 코드뿐 아니라 키 ID·서버 IP·인프라 정보가 전부 공격자의 정찰 자료가 된다
- 위험은 등급으로: 못 막은 것(오리진 평문)도 '왜 낮은지' 근거를 대고 '리스크 수용'으로 문서화한다
