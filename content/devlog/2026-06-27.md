## 1. 남은 보안 구멍 마무리 (#3~#6)

핵심 원칙 하나로 꿸 수 있다 — '클라이언트(브라우저·사용자)가 보낸 값은 절대 그대로 믿지 않는다.' 이메일 소유, 파일 종류, 작성자 이름, 응답이 흘리는 정보까지 전부 이 원칙의 변주다.

### #3. 구독 더블옵트인 — 메일 인프라가 무기가 되는 걸 막기

**취약점** — 이메일 구독 등록(POST /subscribers)이, 입력한 주소가 '정말 본인 것인지' 확인하지 않고 즉시 구독시켰다. 레이트리밋(시간당 10회)만 있었다.

**공격 시나리오** — 공격자가 victim@... (남의 주소)를 구독 등록 → 내 블로그가 그 사람에게 새 글 알림을 보낸다. 받는 사람은 원치 않는 메일을 받고 '스팸 신고'를 누른다. 그러면 발송 서버(SES)의 '발신 평판'이 떨어지고, 심하면 SES 계정이 정지된다. 즉 공격자가 '내 메일 발송 인프라'를 남을 괴롭히는 무기로, 동시에 내 평판을 깎는 데 쓴다.

**조치** — 뉴스레터 표준인 '더블옵트인(double opt-in)'을 넣었다. 등록하면 즉시 구독시키지 않고 '확인 메일'만 한 통 보낸다. 본인이 메일 속 링크를 눌러야 confirmed=True가 되고, 새 글 알림은 '확인된 사람'에게만 간다. 또 등록 응답을 신규/기존 구분 없이 똑같게 만들어, '이 주소가 이미 구독 중인지'도 노출하지 않게 했다.

> **🔎 비유** 회원 명부에 남의 이름을 함부로 못 올리게 하는 것. '이 주소 본인 맞으면 링크 눌러 확인하세요' — 확인 전에는 정기 우편을 안 보낸다.

> **🛠 전문가 노트** DB에 confirmed 컬럼 추가(마이그레이션). 기존 구독자는 정책 이전 가입자라 True로 백필(grandfather). 알림 발송 쿼리에 WHERE confirmed=true 추가가 핵심 방어선. 확인 토큰은 기존 이메일 인증용 JWT(purpose 구분)를 재사용. 실제로 '구독→확인메일 토큰 추출→확인→그 후에만 알림 수신'까지 자동 테스트로 검증.

### #4. 업로드 — 클라이언트가 말하는 '파일 종류'를 믿지 않기

**취약점** — 이미지 업로드가 '파일 종류(content-type)'와 '확장자'를 둘 다 클라이언트가 보낸 값 그대로 믿었다. 둘 다 얼마든지 위조할 수 있다. 그래서 .html이나 .svg 파일을 '이미지인 척' 올릴 여지가 있었다.

**공격 시나리오** — 악성 스크립트가 든 HTML/SVG 파일을 'image/png'라고 속여 업로드 → 저장된 파일이 .html로 남고, 그 주소를 누군가 열면 브라우저가 스크립트를 실행(XSS). (※ 라이브 S3에서는 저장 시 종류를 이미지로 고정해 실행은 막혀 있었지만, '클라이언트 말을 믿는' 설계 자체가 위험의 씨앗이다.)

**조치** — 클라이언트가 보낸 라벨을 아예 안 믿고, 파일 '내용의 맨 앞 바이트(매직 바이트, 파일 시그니처)'로 진짜 종류를 판별했다. PNG/JPEG/GIF/WebP만 통과시키고(HTML·SVG는 거부), content-type과 확장자를 그 판별 결과에서 새로 만들어 붙인다. 사용자가 보낸 파일명은 아예 쓰지 않는다(경로 조작·실행 가능한 확장자 차단).

> **🔎 비유** 봉투에 '사진'이라 적혀 있어도 믿지 말고, 봉투를 열어 안의 내용물을 직접 확인하는 것. 글자(라벨)가 아니라 실물(바이트)로 판단한다.

> **🛠 전문가 노트** 매직 바이트 = 파일 첫 몇 바이트의 고정 패턴. PNG는 89 50 4E 47…, JPEG는 FF D8 FF, GIF는 'GIF87a/89a', WebP는 'RIFF…WEBP'. SVG는 사실상 텍스트(XML)라 스크립트를 품을 수 있어 이미지 화이트리스트에서 제외. 검증: PNG 내용을 evil.html 이름+text/html로 위조 업로드 → 서버가 .png로 정규화해 저장(공격 무력화), HTML 내용을 image/png로 위조 → 400 거부.

### #5. 가입 — '그 사람 회원이에요?'에 답하지 않기 (계정 열거 차단)

**취약점** — 회원가입에서 이미 있는 이메일을 넣으면 409 '이미 가입된 이메일'이라고 알려줬다. 공격자가 이메일 목록을 하나씩 넣어보며 '어떤 주소가 우리 사이트 회원인지'를 알아낼 수 있었다(계정 열거, enumeration). 로그인·비번찾기는 일부러 안 흘리게 만들어 놨는데 가입만 흘렸다 — 일관성 결여.

**공격 시나리오** — 유출된 이메일 리스트를 대입 → 409가 뜨는 주소 = '여기 회원' 확인 → 표적 피싱·크리덴셜 스터핑의 사전 정보로 악용.

**조치** — 신규/기존 구분 없이 항상 똑같은 202 응답('확인 메일을 보냈어')을 준다. 실제 안내는 메일로만 간다 — 신규는 인증 메일, 기존(미인증)은 인증 메일 재발송, 기존(인증완료)은 '이미 가입된 계정' 안내 메일. HTTP 응답만으로는 그 주소의 가입 여부를 1도 알 수 없다(비번찾기와 동일한 패턴).

> **🔎 비유** '이 집에 김XX 사세요?'라고 물으면, 문 앞에서 네/아니오를 말하지 않고 '확인해서 우편으로만 알려드릴게요'라고 일관되게 답하는 것. 집 안 사정은 그 집 우편함(=본인 메일)에만 들어간다.

### #6. 댓글 작성자 사칭 — 로그인하면 이름은 계정이 정한다

**취약점** — 댓글 작성자 이름이 자유 입력이라, 로그인한 사용자도 'admin'이나 블로그 주인 이름으로 댓글을 달 수 있었다(사칭).

**조치** — 로그인 사용자는 작성자명을 '계정 이름(이메일 앞부분)'으로 서버가 강제한다(클라이언트가 보낸 이름은 무시). 익명 댓글만 자유 입력을 유지한다. 화면도 로그인하면 이름 입력칸을 숨기고 고정된 계정명을 보여준다.

> **🔎 비유** 회원은 출입 시 자동으로 명찰(계정명)이 박히고, 비회원만 손으로 이름표를 쓴다. 회원이 남의 명찰을 위조할 수 없다.

(#6은 기술적 '침투'는 아니고 사회공학 영역이지만, 신뢰를 무너뜨리는 사칭이라 막아두는 게 맞다.)

## 2. AI 글 초안 기능 — 따로 깊게 점검 (A~D)

거친 메모를 정돈된 글 초안으로 바꿔주는 'AI 초안' 기능은 ① 비용(서버의 Claude 키로 호출하면 내 돈) ② 외부 호출(사용자가 자기 키(BYOK)로 다른 LLM 주소를 부를 수 있음)이 얽혀 있어 보안 결이 다르다. 기본기(권한 게이팅, 키 암호화 저장, 이전에 막은 SSRF, 모델 등급 제한)는 이미 탄탄했고, 그 위에 4가지를 더 보탰다.

### A. 레이트리밋 우회 — X-Forwarded-For 위조 (가장 중요) 🔴

**취약점** — 서버가 '진짜 방문자 IP'를 알아낼 때 X-Forwarded-For(XFF) 헤더의 '맨 앞' 값을 썼다. 그런데 XFF 맨 앞은 클라이언트가 마음대로 적어 넣을 수 있는 값이다.

**공격 시나리오** — 공격자가 요청마다 XFF 맨 앞에 매번 다른 가짜 IP를 적으면, 서버는 '매번 다른 사람'으로 착각한다 → 로그인 무차별 대입 방어, AI 비용 캡 등 '모든 레이트리밋'이 통째로 무력화된다.

**조치** — 맨 앞이 아니라 '맨 뒤' 값을 쓰게 바꿨다. CloudFront(우리가 신뢰하는 정문 프록시)는 진짜 방문자 IP를 헤더 '맨 뒤'에 직접 덧붙인다. 그러니 맨 뒤 = 프록시가 실제로 관측한 IP라 위조가 안 된다. 여기에 인프라 짝(EC2의 8000 포트를 CloudFront에서 오는 것만 허용 → 정문 우회 차단)이 합쳐져야 완전한데, 그건 이미 돼 있었다.

> **🔎 비유** 택배 송장의 '최초 발송인' 칸은 누구나 거짓으로 적을 수 있다. 믿을 수 있는 건 '바로 직전에 거쳐 간 우체국(CloudFront)이 직접 찍은 도장' — 그게 헤더의 맨 뒤 칸이다.

> **🛠 전문가 노트** 이 한 줄(맨 앞→맨 뒤)이 모든 엔드포인트의 레이트리밋 신뢰성을 좌우한다. 단, 백엔드를 프록시 거치지 않고 직접 칠 수 있으면 헤더째 위조되므로, 방화벽(보안그룹)으로 직접 접근을 막는 인프라 조치가 반드시 짝이 된다.

### B. LLM 호출 타임아웃 — 느린 응답으로 일꾼 묶기(DoS) 🟠

**취약점** — BYOK의 'compatible' 종류는 사용자가 '호출할 LLM 주소'를 직접 지정한다. 만약 악의적이거나 아주 느린 서버를 지정하면, 응답을 질질 끌어 우리 서버의 처리 일꾼(worker)이 기본 10분까지 묶인다. 몇 개만 동시에 보내도 백엔드가 마비될 수 있다.

**조치** — 모든 LLM 호출(Claude/OpenAI/Gemini/Cohere)에 타임아웃 60초 + 재시도 1회를 명시했다.

> **🔎 비유** 전화 상대가 말없이 수화기만 붙들고 있으면 1분 뒤 끊어버리는 것. 전화선(일꾼)이 영영 묶이지 않게.

### C. SSRF 재검증 — 검증한 주소가 '나중에' 바뀌는 함정 🟡

**배경** — BYOK 'compatible'의 base_url은 '서버가 직접 요청을 보내는 주소'다. 그냥 두면 승인된 사용자가 이 서버를 발판 삼아 내부망이나 클라우드 메타데이터(169.254.169.254 — 서버 자격증명이 들어있는 위험 주소)를 찌를 수 있다(SSRF). 저장할 때 'https만 + 공인 IP만' 검증은 이미 하고 있었다.

**취약점** — 검증을 '저장 시점'에만 했고, 실제 호출 때는 저장된 주소를 그대로 썼다. 그 사이 DNS가 공인 IP→내부 IP로 바뀌면(DNS rebinding) 우회된다.

**조치** — 호출 직전에 같은 검증을 한 번 더 돌려, 바뀔 수 있는 창(window)을 크게 줄였다. 완전한 최종 방어선은 인프라(IMDSv2 강제)다.

> **🔎 비유** 출입증을 발급할 때만 신원을 보는 게 아니라, 문을 통과하기 직전에 한 번 더 확인하는 것. 그래도 '발급~통과 사이' 위조 가능성은 남아, 금고 자체(IMDSv2)도 잠가둔다.

### D. 유저별 일일 비용 캡 — 서버 돈 새는 것 막기 🟡

**취약점/배경** — 서버 Claude 키로 하는 호출은 '내 돈'이 나간다. 시간당 횟수 제한(10회/시)만 있었고 '하루 총량' 제한이 없었다. A(레이트리밋 우회)와 겹치면 비용 폭주 위험이 커진다.

**조치** — 사용자별로 '하루' 서버키 호출 상한(기본 20회)을 두었다. 새 테이블(ai_usage)에 일별 횟수를 기록하고, 한도 넘으면 호출 전에 막는다. BYOK(사용자 자기 키) 호출은 본인 돈이라 이 캡에서 제외한다.

> **🔎 비유** 무료 샘플(서버 키)은 하루 N개까지만. 자기 돈 내고 쓰는 손님(BYOK)은 제한 없음.

> **🛠 전문가 노트** 한도 초과 시 '비싼 호출을 하기 전에' 429로 차단하도록 순서를 잡았다 — 막힐 거면 돈 쓰기 전에 막아야 의미가 있다. 캡을 가득 채워 두고 호출하면 실제 LLM 호출 없이 즉시 429가 나오는 것까지 검증(비용 0).

## 3. 점검 중 튀어나온 부수 버그 2개

### 3-1. openai 라이브러리 ↔ httpx 버전 충돌

AI 호출에 타임아웃을 넣다가, OpenAI 클라이언트가 '생성 자체가 안 되는' 걸 발견했다. 원인은 라이브러리 버전 궁합 — openai 1.54.3이 내부에서 httpx에 'proxies'라는 인자를 넘기는데, 새 httpx(0.28)가 그 인자를 없애버려 충돌했다. 즉 openai/Grok 등 BYOK 경로는 '실행하면 무조건 에러(502)'인 상태였다(이전부터 있던, 보안과 무관한 기능 버그). openai를 1.55.3으로 올리고 이미지를 재빌드해 해결.

> **🔎 비유** 새 콘센트(httpx)는 둥근 핀을 없앴는데, 옛 플러그(openai)는 둥근 핀을 그대로 꽂으려 한 것. 플러그를 새 규격으로 교체.

### 3-2. 마이그레이션 설정 누락 — '키 보관함'을 지울 뻔

DB 구조 변경 도구(alembic)에 BYOK 키 저장 테이블(llm_credentials)의 모델이 등록되어 있지 않았다. 이 상태로 자동 생성을 돌리면 도구가 그 테이블을 '코드에 없으니 지워라'라고 판단해 삭제 마이그레이션을 만들 수 있었다 — 사용자 키가 통째로 날아갈 위험. 등록 한 줄을 추가해 막았다.

> **🔎 비유** 재고 목록에 '귀중품 금고'를 안 적어둬서, 정리 로봇이 '목록에 없네' 하고 금고를 버리려던 상황. 목록에 한 줄 추가로 방지.

## 4. 프로덕션 백엔드 배포 — 한참 밀려 있던 운영 DB

오늘 작업을 운영 서버(EC2)에 올렸다. 배포는 '코드 압축(tar) → 서버 전송(scp) → 도커 재빌드'이고, 컨테이너가 뜰 때 운영 DB(RDS)에 구조 변경 이력(마이그레이션)이 자동 적용된다.

배포하며 알게 된 사실 — 운영 DB가 생각보다 한참 뒤처져 있었다. 그동안 만든 기능들(외부 LLM 키 BYOK, 글 공개범위 3단계, 유료 등급 is_pro 등)이 '코드는 커밋됐지만 운영엔 미배포'였던 것. 이번 배포가 그 밀린 마이그레이션 6개를 순서대로 한 번에 적용하며 그 기능들 + 오늘의 보안 작업까지 전부 라이브로 만들었다. 기존 데이터(글·구독자)는 그대로 유지됐다.

> **🔎 비유** 창고(운영 DB)의 도면이 옛날 버전이었는데, 이번 배송과 함께 그동안 밀려 있던 도면 6장이 순서대로 자동 시공된 것. 안에 있던 물건은 안 건드리고.

**검증** — health·status 정상(backend/database/mail 전부 ok), DB 버전이 최신(head), 구독 등록이 '더블옵트인' 응답을 주는 것까지 라이브에서 확인. 배포용 테스트 데이터는 정리.

주의로 남긴 것: 프론트엔드는 git push로 자동 배포(GitHub Actions)되는데, 이건 토큰 인증이 필요해 직접 했다. 백엔드만 먼저 최신이라, push 전까지는 프론트가 구버전 → 신규 구독자의 '확인 링크' 화면이 아직 없는 상태였다. (push로 마무리.)

## 5. 오늘 익힌 핵심 개념

- 신뢰 경계(trust boundary): 클라이언트가 보낸 값(파일 종류·이름, 작성자, XFF 헤더)은 '서버가 직접 확인'하기 전엔 안 믿는다 — 오늘 거의 모든 조치의 공통 뿌리
- 정보 누출 차단: 응답·에러 메시지로 '계정 존재' 같은 걸 흘리지 않기(가입 enumeration). 비번찾기·로그인과 일관되게
- 메일 인프라 보호: 더블옵트인으로 발신 평판(SES)을 무기화당하지 않기
- 자원·비용 보호: 외부 호출 타임아웃(DoS), 일일 캡(비용), SSRF 재검증
- 레이트리밋의 전제: '클라이언트 IP를 정확히 식별'해야 의미가 있다(프록시 뒤에선 맨 뒤 + 직접 접근 차단)
- 배포 일관성: 프론트/백엔드 버전이 맞아야 한다. 운영 DB가 코드보다 뒤처져 있을 수 있으니 배포 시 마이그레이션 적용을 늘 확인
