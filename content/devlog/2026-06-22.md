# 블로그 만들기 #2 — 통합 포털 · 서비스 상태 · AI 글 초안

## 0. 오늘 요약

- 통합 사이트(포털): 루트(/)에 로비를 두고 블로그(/blog)·상태(/status)로 갈라지는 입구 신설.
- 서비스 상태 + 업타임: 백엔드/DB/메일 점검 + 가동률 %를 서비스별 날짜 막대로 표시.
- AI 글 초안: 거친 메모 → 정돈된 글 구조 마크다운. 코드 완료, API 키만 넣으면 동작.
- Docker 전체 컨테이너화(오늘 메인): `docker compose up` 한 줄로 4개 컨테이너 전체 기동.

## 1. 통합 사이트(포털)

비유: 회사 건물에 들어가면 바로 사무실이 아니라 로비 안내판에서 부서별 층을 보고 가듯, 사이트 루트에 '로비(포털)'를 두고 블로그/상태정보로 안내하는 구조.

### 무엇을 했나

- / = 통합 랜딩(블로그·상태정보 카드 2개), 기존 블로그 홈은 /blog 로 이동.
- 라우트 재배치: /blog, /blog/posts/:id(+/edit), /blog/new, /status, /login, /register.
- 라우팅을 옮기면 내부 링크가 깨지므로 글 링크·'목록으로'·작성 후 이동 등을 전부 함께 수정.

### 실무 포인트

- URL을 /blog 아래로 모으면 이후 기능 추가(/shop, /docs 등) 시에도 구조가 깔끔하다.

## 2. 서비스 상태 페이지 + 업타임

비유: 지하철역의 '에스컬레이터 가동중/점검중' 전광판, 또는 claude status / github status 페이지.

### 무엇을 했나

- 백엔드 /status: backend·database·mail 점검 + 통계(글 수·구독자 수) 반환.
- 메일 점검은 Mailpit SMTP(1025) 소켓 연결로, DB는 select 1 + count 쿼리로.
- 업타임: status_checks 테이블에 1분마다 백그라운드 스레드가 자가 점검을 1줄씩 기록.
- /status/history?days=N: 일별로 '정상 점검 ÷ 전체 점검' 비율을 서비스별로 집계.
- 프론트: 백엔드/DB/메일 각각 막대(초록=정상, 노랑=일부장애, 빨강=장애, 회색=데이터없음) + 전체 %.

### 실무 포인트 (정직한 한계)

- 자기 자신이 죽으면 자기가 기록을 못 남긴다 → 우리 업타임은 '돌고 있을 때 건강했나' 기준.
- 진짜 다운타임 측정은 '밖에서 찔러보는 외부 감시자'가 필요(inside-out vs outside-in 모니터링).
- 외부 모니터는 이후 AWS 단계(Lambda + 크론)에서 진짜로 붙일 수 있음.

## 3. AI 글 초안 생성 (코드 완료 / 키 대기)

비유: 회의 후 휘갈긴 메모를 비서가 깔끔한 보고서 골격으로 정리해 주는 것.

### 무엇을 했나

- 백엔드 POST /ai/draft: 메모를 받아 Claude API로 '제목·소제목·초안' 마크다운 생성.
- 공식 anthropic SDK 사용, 모델은 config.ai_model(기본 claude-opus-4-8, .env로 교체 가능).
- 프론트: 글쓰기 페이지에 'AI 초안' 박스(메모 입력 → 생성 → 첫 # 제목은 제목칸, 나머지는 본문).

### 실무 포인트

- 비용 보호: 유료 API라 아무나 호출 못 하게 로그인 필수로 제한(봇이 비용을 태우는 것 방지).
- 시크릿 관리: 키는 .env(gitignore)에만, 코드/커밋 절대 금지. 모델명도 env로 분리.
- 현재 키 미설정 → 호출 시 친절한 503 안내. $5 충전 + 키 입력이면 즉시 동작.

## 4. Docker 전체 컨테이너화 (오늘 메인)

비유: 이삿짐을 규격 컨테이너에 담으면 트럭→배→기차 어디든 그대로 옮겨지듯, 앱을 컨테이너로 싸면 '내 노트북에서 되던 것'이 AWS 서버에서도 그대로 된다.

### 핵심 개념

- 이미지 = 냉동만두(설계도/스냅샷), 컨테이너 = 쪄낸 만두(실행 인스턴스). 클래스 vs 인스턴스와 같음.
- 레이어 캐시: Dockerfile 한 줄=필름 한 장. 안 바뀐 아래 층은 재사용 → requirements 설치를 코드 복사보다 위(아래 층)에 둠.
- 네트워킹: 컨테이너 안 localhost는 '자기 자신'. DB/메일은 compose 서비스 이름(db, mailpit)으로 접속. AWS에선 이 자리에 RDS 주소가 들어감.
- 볼륨: named volume(postgres_data)=USB 같은 영속 저장소 → 컨테이너 지워도 데이터 유지. bind mount(./backend:/app)=공유폴더(코드 즉시 반영).
- healthcheck: DB는 컨테이너가 켜져도 몇 초간 연결을 못 받음 → pg_isready로 '준비 완료' 확인 후 백엔드 기동(경합 방지).

### 1단계 — 백엔드 컨테이너

- backend/.dockerignore: .venv·uploads·.env 제외(경량화 + 시크릿 보호).
- compose backend: env_file(없던 .env) 제거 → environment로 DATABASE_URL=@db, SMTP_HOST=mailpit 주입.
- db healthcheck(pg_isready) + depends_on: service_healthy.
- command: 'alembic upgrade head && uvicorn ...' → 시작 시 마이그레이션 자동.

### 2단계 — 프론트 컨테이너 + 전체 한 방

- 멀티스테이지 Dockerfile: node로 빌드(dist 생성) → nginx가 정적 서빙(최종 이미지엔 node/소스 없음).
- nginx.conf SPA 폴백(try_files … /index.html): /blog 등 새로고침해도 404 안 남.
- 포트 5173:80(nginx 80 → 호스트 5173) → CORS origin(localhost:5173) 유지, 백엔드 안 고침.
- AWS S3+CloudFront 정적 배포와 동일한 구조 → 이후 이전이 수월.

## 6. 자주 쓰는 Docker 명령

- docker compose up -d --build   # 전체 빌드+기동(백그라운드)
- docker compose ps              # 상태 확인
- docker compose logs -f backend # 로그 보기
- docker compose down            # 정지(데이터 유지)
- docker compose down -v         # 데이터까지 삭제(주의)
- docker compose up -d --build frontend  # 프론트 코드 수정 후 다시 굽기

주의: 현재 셸은 docker 그룹 미반영 → 실제로는 sg docker -c "..."로 감싸 실행. 백엔드 포트 종료는 fuser -k 8000/tcp (pkill은 자기 셸까지 죽임).
