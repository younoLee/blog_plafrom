# 블로그 만들기 #9 — 또 검은 화면, 이번엔 진짜 크래시 + 전체 코드 감사

> 웹/클라우드 입문자가 읽어도 이해되도록 — 버그를 '끝까지 파는' 사고 과정과 보안 점검 위주로.

## 0. 오늘 한 일 한눈에

어제는 '느려서' 생긴 문제였는데, 오늘은 비슷해 보이지만 원인이 다른 '진짜 크래시'를 잡았다. 유효하지 않은 키로 AI 초안을 부르면 앱이 통째로 하얗게(검게) 죽는 버그였다. 원인을 끝까지 파보니 '브라우저 자동번역'이라는 의외의 범인이 나왔고, 같은 부류 버그를 앱 전체에서 찾아 고친 뒤, 이 김에 프론트·백엔드 코드를 통째로 한 번 더 보안 감사했다.

- 크래시 추적: 'insertBefore ... not a child' 에러 — 범인은 브라우저 자동번역
- 같은 결함 일괄 수정: 버튼 3개 구조 안정화 + 본문 메모이즈 + 전역 안전망
- 전체 감사: 라우터·인증·SSRF·XSS·토큰 등 프론트/백엔드 전수 점검(치명·고위험 0)
- 운영 견고성 4건 보강 + ESLint 경고 0으로 정리
- 배포 + 라이브 검증(레이트리밋 429, 번들 안 가드 확인)

> **🔎 비유** 어제가 '배달이 느려 손님이 가버리는 문제'였다면, 오늘은 '주방에 몰래 들어와 접시를 옮겨놓는 유령(번역기) 때문에 직원(React)이 헛손질하다 넘어지는 문제'를 잡고, 주방 전체 안전점검까지 한 날이다.

## 1. 크래시 추적 — 'insertBefore' 에러의 진짜 범인

에러 메시지는 이랬다: "Failed to execute 'insertBefore' on 'Node': 새 노드를 끼워넣을 기준 노드가 이 노드의 자식이 아니다." 화면을 그리는 React가 DOM(화면 구조)을 만지다 '어? 분명 여기 있어야 할 조각이 다른 데 가 있네' 하고 넘어진 것이다.

### 범인 = 브라우저 자동번역

**원인** — 디스코드 같은 인앱 브라우저나 구글 번역은 화면의 '글자 조각(텍스트 노드)'을 몰래 <font> 태그로 감싼다. 그러면 React가 기억하던 '글자 조각의 위치'가 어긋나고, 나중에 그 옆에 뭔가를 끼워넣거나 지우려 할 때 '이건 내 자식이 아닌데?' 하며 크래시난다.

**취약했던 모양** — 특히 ① 버튼 안에서 [아이콘 + 맨글자]를 통째로 바꿔치기(로딩↔완료), ② 에러/완료 메시지가 버튼 옆에 생겼다 사라지는 구조가 이 함정에 약했다. 유효하지 않은 키 → 에러 표시로 전환되는 그 순간이 정확히 트리거였다.

> **🔎 비유** React는 '몇 번 서랍에 뭘 넣었다'를 외워둔다. 그런데 번역기가 몰래 서랍 전체를 다른 칸으로 옮기면, React가 원래 칸을 열었을 때 물건이 없어 당황하고 작업이 멈춘다.

> **🛠 전문가 노트** 핵심 단서는 '크롬에선 멀쩡, 디스코드에선 크래시'였다. 같은 코드가 한 곳에서만 죽으면 코드 자체보다 '그 환경이 뭘 다르게 하나'를 의심해야 한다 — 여기선 인앱 브라우저의 번역/DOM 변형이 그 차이였다.

## 2. 수정 — 구조 안정화 + 전역 안전망 (3중 방어)

**① 구조 고정** — 흔들리던 버튼들(AI 초안·상태 새로고침·구독)을 '항상 [아이콘][<글자칸>]을 유지하고 내용만 바꾸는' 구조로 바꿨다. 맨글자(텍스트 노드)를 칸(span)으로 감싸면 번역기가 건드려도 위치가 안 어긋난다. 에러/완료 메시지도 '항상 자리하는 칸'으로 만들어, 생겼다 사라지며 구조가 흔들리는 걸 없앴다.

**② 본문 메모이즈** — 글 본문(마크다운)은 댓글이나 구독 상태가 바뀔 때마다 괜히 다시 그려지고 있었다. '내용이 바뀔 때만 다시 만들기(메모이즈)'로 묶어, 번역된 본문을 불필요하게 재조정하다 깨지는 걸 막았다.

**③ 전역 안전망** — 번역기의 DOM 변형은 우리가 막을 수 없다. 그래서 React가 쓰는 두 DOM 함수(노드 삽입·삭제)를 감싸 '대상이 진짜 자식이 아니면 조용히 건너뛰게' 했다. 정상 동작엔 영향이 없고, 어긋난 경우만 크래시 대신 넘어간다 — 우리가 놓친 곳에서 번역이 사고를 쳐도 앱이 안 죽는다.

> **🔎 비유** ①②는 '서랍을 잘 정리해 안 흔들리게', ③은 '그래도 누가 흔들면, 직원이 빈 칸을 열어도 넘어지지 말고 그냥 넘어가도록' 가르친 것. 둘을 같이 해야 안전하다.

> **🛠 전문가 노트** ①처럼 코드를 안정시키는 게 우선이고, ③ 전역 안전망은 '우리가 못 보는 외부 변형'에 대한 보험이다. 외부(번역기·확장프로그램)는 통제 불가라, 통제 가능한 것(구조)부터 고치고 나머지는 안전망으로 받친다.

## 3. 이 김에 — 프론트·백엔드 전체 코드 감사

버그 하나가 비슷한 게 더 있을까 의심하게 만들었고, 그 김에 코드 전체를 보안 관점으로 다시 훑었다. 다행히 치명·고위험은 0건이었고(그간 여러 번 점검한 덕), 운영 견고성 위주로 네 가지를 보강했다.

**A. 강제 로그아웃 버그** — 내 정보를 불러오는 코드가 '서버가 잠깐 5xx로 흔들리기만 해도' 로그인 토큰을 지워버렸다 → 서버가 깜빡할 때마다 사용자가 강제 로그아웃. '정말 만료(401)일 때만' 지우게 고쳤다.

**B. 동시 가입 충돌** — 같은 이메일로 동시에 가입 요청이 들어오면 DB 충돌로 500 에러가 났다. 이를 잡아 평범한 응답을 유지(계정 존재 여부도 안 흘림)하게 했다.

**C. 상태 점검 DoS** — 로그인 없이 부를 수 있는 '상태 점검' 주소가 매 호출마다 메일 서버에 2초씩 연결을 시도했다. 누가 마구 부르면 서버 일꾼이 묶인다 → 분당 횟수 제한을 걸었다(실제로 한도 초과 시 막히는 것 확인).

**D. 메일 제목 안전화** — 알림 메일 제목에 글 제목(사용자 입력)이 들어가는데, 줄바꿈을 제거해 메일 헤더 조작 여지를 없앴다.

그리고 '이미 잘 돼 있어 안 건드린 것'도 확인했다:

- SSRF 방어: 서버가 호출하는 외부 주소는 https + 공인 IP만 허용(내부망·메타데이터 차단)
- 마크다운 XSS: 글 본문은 raw HTML을 안 그리고 위험한 링크를 걸러냄 → 스크립트 주입 차단
- 권한/숨김: 남의 비공개 글·댓글은 '존재 자체를 404'로 숨김, 가입/로그인은 계정 존재를 안 흘림
- 토큰: 비번 재설정·차단 시 기존 로그인 전부 무효화, 기본 비밀키 없으면 서버가 아예 안 뜸

> **🔎 비유** 집을 고치다 '비슷한 데가 더 있나' 싶어 집 전체를 한 바퀴 점검한 것. 큰 구멍은 없었고, 헐거운 나사 몇 개(견고성 4건)만 조였다.

## 4. 마무리 — 경고도 0으로

코드 품질 검사기(ESLint)가 내 직전 수정에서 지적 3건을 냈다. '에러를 다시 던질 때 원래 원인을 같이 달아라(디버깅 단서 보존)', '메모이즈가 의존하는 값을 정확히 적어라' 같은 좋은 지적이라 그대로 반영했다. 이제 경고·에러 0건.

> **🛠 전문가 노트** 경고를 '나중에'로 미루면 쌓여서 진짜 문제를 가린다. 0으로 유지하면 새로 생긴 경고가 바로 눈에 띈다.

## 5. 배포 + 라이브 검증

프론트는 git push로 자동 배포, 백엔드는 코드 전송 후 재빌드. 어제 '말만 다 했고 실제론 절반'이었던 교훈대로, 이번엔 라이브에서 직접 확인했다.

**검증** — 상태 점검 주소를 35번 연속 호출 → 30번 성공 후 5번 차단(429)으로 레이트리밋 작동 확인. 배포된 화면 파일(번들) 안에 전역 안전망 코드가 들어간 지문도 확인. 백엔드 컨테이너에 새 코드 반영 확인.

> **🔎 비유** 자판기 새로 고치고 '동전 35번 넣어 30번째부터 막히는지'까지 눌러본 것 — 설명서가 아니라 실물로 확인.

## 6. 오늘 익힌 핵심 개념

- 환경 차이로 좁히기: '여기선 되고 저기선 안 됨'은 코드보다 '환경이 뭘 다르게 하나'를 가리킨다
- 외부는 통제 불가: 번역기·확장프로그램의 DOM 변형은 못 막으니, 구조를 안정화 + 전역 안전망으로 받친다
- 불필요한 재렌더 줄이기(메모이즈): 안 바뀐 부분을 다시 안 그리면 성능 + 안정성 둘 다 이득
- 정기 감사: 버그 하나가 같은 부류를 의심하게 한다 — 그 김에 전체를 훑으면 잠복 결함을 잡는다
- 견고성도 보안이다: 강제 로그아웃·동시성 충돌·무인증 DoS는 '취약점'은 아니어도 사용자 신뢰를 깎는다
- 경고 0 유지: 새 경고가 묻히지 않게 하는 위생 습관
