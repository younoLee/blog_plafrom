## 1. AI 초안 — '코드 출력 막기' (시스템 프롬프트 가드레일)

AI 초안은 '거친 메모 → 정돈된 블로그 글 구조'로 바꿔주는 도구다. 그런데 메모에 '도커 설정 보여줘', '파이썬 코드도 넣어줘' 같은 말이 들어가면 AI가 실제 코드 블록을 만들어버린다. 이 도구의 목적은 '글의 뼈대 잡기'지 코드 생성이 아니라서, 코드는 사람이 직접 검토해 넣는 게 맞다. 그래서 AI에게 주는 '지시문(시스템 프롬프트)'에 코드 금지 규칙을 넣었다.

**조치** — 시스템 프롬프트에 규칙 추가 — 코드 블록(``` ~~~)·인라인 코드·명령어·설정 예시를 절대 만들지 말고, 코드가 필요한 자리는 '[여기에 코드 예시를 직접 넣어주세요]' 플레이스홀더로만 표시하라.

**검증** — 코드를 일부러 요구하는 메모(도커 컴포즈·파이썬 함수·git 명령어 정리해줘)를 실제 Claude로 호출 → 출력에 코드 블록 0개, 인라인 코드 0개. 코드 자리는 전부 플레이스홀더로 대체됨. 운영 서버에서도 동일하게 0개 확인.

> **🔎 비유** 요리 레시피를 '구조만' 잡아주는 비서에게 '재료 손질 영상은 네가 찍지 말고, 영상 넣을 자리만 비워둬'라고 정해두는 것. 비서는 글의 흐름만 만들고, 실제 코드(영상)는 전문가가 직접 채운다.

> **🛠 전문가 노트** 시스템 프롬프트는 LLM의 '행동 헌법' 같은 것 — 사용자 입력보다 우선한다. 다만 100% 강제는 아니라서(LLM은 확률적), '규칙으로 막고 + 실제 호출로 확인'까지 해야 신뢰할 수 있다. 오늘은 코드를 노골적으로 요구해도 안 뱉는 것까지 실측했다.

## 2. 사용량 제한 추가 — 월간 캡 + '남은 횟수' 보여주기

어제 '하루 20회' 캡(서버 키 호출 비용 폭주 방지)을 넣었는데, 매일 조금씩 쓰면 한 달 누적은 막지 못한다. 그래서 '한 달' 한도를 2차 방어선으로 추가하고, 사용자가 한도를 체감하도록 글쓰기 화면에 '남은 횟수'를 띄웠다.

### 2-1. 월간 캡 (2차 방어선)

**조치** — 월 한도(기본 200회)를 일일 캡과 별개로 추가. 호출 전에 '오늘 한도'와 '이번 달 한도'를 둘 다 확인하고, 둘 중 하나라도 넘으면 막는다(429). 서버 키(Claude) 호출만 해당하고, 사용자 자기 키(BYOK)는 본인 돈이라 제외.

> **🛠 전문가 노트** DB 설계 포인트 — 새 테이블을 안 만들었다. 어제 만든 '일별 사용 기록' 테이블(ai_usage)이 이미 있으니, 월간은 그걸 '이번 달 1일~오늘'로 합(SUM)만 내면 된다. 같은 데이터를 두 각도로 보는 것 → 마이그레이션(DB 구조 변경) 불필요, 데이터 중복 없음.

> **🔎 비유** 일일 캡은 '하루 통행료 상한', 월간 캡은 '한 달 통행료 상한'. 매일 조금씩 다녀도 월 상한에 걸리면 멈춘다. 둘 다 같은 '통행 기록' 장부 하나로 계산한다 — 장부를 새로 팔 필요가 없다.

### 2-2. '남은 횟수' 표시 (UX)

**조치** — 사용량을 알려주는 조회 API(GET /ai/usage)를 새로 만들고, 글쓰기 화면이 진입할 때와 초안 생성 직후에 그걸 불러와 '서버 모델 남은 횟수 · 오늘 N/20 · 이번 달 M/200'을 보여준다(소진되면 빨강).

> **🔎 비유** 커피 쿠폰에 '오늘 몇 잔 남음 · 이번 달 몇 잔 남음'을 도장 옆에 적어두는 것. 손님이 한도를 미리 알고 쓴다.

## 3. 모바일 화면 정리 (반응형)

폰처럼 좁은 화면에서 화면이 깨지는지 점검했다. 다행히 대부분 페이지는 이미 화면 크기에 맞춰 바뀌도록 만들어져 있었고, 딱 한 곳 — 상단 헤더 — 만 문제였다. 로그인하면 헤더에 버튼(구독·관리자·설정·글쓰기·로그아웃·테마)이 줄줄이 들어가는데, 이게 한 줄에 욱여넣어져 좁은 화면에서 넘쳤다.

**조치** — ① 공용 버튼의 좌우 여백을 모바일에서 살짝 줄이고(데스크탑은 그대로), ② 헤더의 버튼 묶음이 넘치면 자연스럽게 '다음 줄로 줄바꿈(flex-wrap)'되게 했다. 새 버튼을 숨기거나 햄버거 메뉴를 새로 만들지 않고, 가장 단순하게 해결.

> **🔎 비유** 좁은 책상에 물건이 많으면, 억지로 한 줄로 밀어 떨어뜨리지 말고 '아랫줄에 한 칸 내려놓게' 하는 것. 데스크탑(넓은 책상)에선 원래대로 한 줄.

> **🛠 전문가 노트** 반응형은 화면 폭에 따라 규칙을 바꾸는 것. 여기선 Tailwind의 'sm:' 접두사를 썼다 — 평소(모바일)엔 좁은 여백, 일정 폭 이상(sm↑)이면 원래 여백. 즉 데스크탑 모양은 1픽셀도 안 바뀌고 모바일만 개선된다.

## 4. 운영 서버 배포

위 변경을 커밋하고 운영(라이브)에 올렸다. 백엔드와 프론트엔드는 배포 방식이 다르다.

**백엔드** — 코드 압축(tar) → 서버(EC2) 전송(scp) → 도커 재빌드. 이번엔 DB 구조 변경(마이그레이션)이 없어서 운영 DB(RDS)는 그대로다. 운영 서버의 비밀 설정 파일(.env)·업로드 이미지는 건드리지 않게 코드만 갈아끼웠다.

**프론트** — git push 한 번으로 GitHub Actions가 자동 빌드·배포(S3 + CloudFront). 라이브 화면 파일(번들)에 새 기능 문자열('남은 횟수', 사용량 조회 호출)이 들어간 것까지 확인.

**검증** — 운영 컨테이너 재기동 확인, 컨테이너 안에 새 코드 반영 확인, 새 API(/api/ai/usage)가 라이브에서 응답(로그인 필요라 401 = 정상 도달), 라이브 health 200.

> **🔎 비유** 백엔드 배포는 '주방 설비를 새 걸로 교체', 프론트 배포는 '손님이 보는 메뉴판을 새로 인쇄'. 둘 다 바꿔야 손님이 새 기능을 제대로 쓴다.

## 5. 운영 AI 키 활성화 — 그리고 'env 파일' 함정

운영 서버에서 AI 초안이 실제로 작동하려면 'API 키'가 운영 서버에 있어야 한다. 키 2개가 필요하다.

- ANTHROPIC_API_KEY — 서버가 대신 Claude를 부를 때 쓰는 키(기본 초안 기능). 없으면 초안 503
- LLM_ENCRYPTION_KEY — 사용자가 맡긴 자기 키(BYOK)를 DB에 암호화 저장할 때 쓰는 자물쇠. 없으면 BYOK 등록 503

### 오늘의 실수 → 교훈: 로컬 .env ≠ 운영 .env

**무슨 일** — '루트 .env에 키 넣었어'라고 했는데 운영엔 안 들어가 있었다. 알고 보니 건드린 건 '내 PC의 프로젝트 폴더 .env'(개발용)였고, 운영 서버(EC2)의 .env는 완전히 다른 파일이었다.

**왜 그런가** — 내 PC의 .env는 비밀이라 git에 안 올라가고(.gitignore), 배포 압축(tar)에서도 일부러 제외한다 → 그래서 push해도 배포해도 운영엔 절대 안 간다. 운영 비밀은 운영 서버에서 직접 넣어야 한다.

**또 하나의 함정** — '.env 마지막 줄'을 복사해 넣으려 했는데, 마지막 줄은 LLM_ENCRYPTION_KEY였다. 필요한 Claude 키(ANTHROPIC)는 가운데 줄이라, 자칫 '이름은 ANTHROPIC인데 값은 암호화 키'가 들어갈 뻔했다. 키는 '이름 ↔ 값'을 짝 맞춰 넣어야 한다.

**해결** — 운영 서버(EC2)의 .env에 두 키를 각자 이름에 맞는 값으로 직접 추가(키 값은 사용자가 입력, AI는 보지 않음) → env만 다시 읽도록 컨테이너 재기동(빌드 불필요). 첫 줄 SECRET_KEY(로그인 토큰 서명용)는 운영이 예전에 자체 생성한 값이라 건드리면 기존 로그인이 다 풀려 — AI 두 개만 추가.

> **🔎 비유** 내 집 열쇠 복사본(로컬)을 아무리 만들어도 회사 금고(운영)는 안 열린다. 회사 금고 비밀번호는 회사에 가서 직접 입력해야 한다. 그리고 '현관 열쇠 칸'에 '자동차 열쇠'를 꽂으면 안 되듯, 키는 이름과 값을 정확히 짝지어야 한다.

> **🛠 전문가 노트** 12요소 앱 원칙 — '설정(비밀)은 코드와 분리해 환경(서버)에 둔다.' 그래서 로컬/운영이 의도적으로 다르다. 헷갈리기 쉬운 만큼, 운영 비밀은 항상 운영 서버에서 확인하는 습관이 안전하다.

## 6. 라이브에서 직접 시험 (e2e)

준비가 끝났으니 실제로 써봤다. 운영 사이트에 로그인 → 글쓰기 → AI 초안 생성. 화면엔 '남은 횟수'가 뜨고, 초안이 제목·본문에 채워졌다.

**서버측 검증** — 운영 DB의 사용 기록(ai_usage)이 0 → 증가하는 것을 직접 확인. 일일 한도와 월간 한도가 '동시에' 차감되는 것까지 맞았다. 즉 화면의 '남은 횟수'가 실제 DB 값과 연결돼 살아 움직였다.

> **🔎 비유** 새 자판기를 설치만 한 게 아니라, 동전을 넣고 버튼을 눌러 음료가 나오고 '남은 재고' 숫자가 줄어드는 것까지 눈으로 확인한 것.

## 7. 오늘 익힌 핵심 개념

- 시스템 프롬프트 = AI의 행동 규칙(사용자 입력보다 우선). 단, 확률적이라 '규칙 + 실측'으로 보증
- 다층 한도: 시간당(폭주) · 일일(하루 총량) · 월간(누적) — 같은 기록 하나를 다른 창으로 보면 새 테이블 없이 구현 가능
- UX는 기능의 일부: 한도가 있으면 '남은 양'을 보여줘야 사용자가 납득하고 쓴다
- 반응형은 '규칙 분기' (모바일/데스크탑) — 가장 단순한 해법부터(여백·줄바꿈), 햄버거 메뉴는 정말 필요할 때
- 설정/비밀은 환경에 둔다(12요소): 로컬 .env와 운영 .env는 다른 파일이며, 운영 비밀은 운영에서 직접 넣는다
- 키는 이름↔값 짝맞춤: 같은 파일의 다른 줄을 헷갈리면 엉뚱한 비밀이 들어간다
